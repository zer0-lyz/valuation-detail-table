## 教训1：插入行后公式断裂（反复出现，最高频）

**问题**：Python脚本通过`ws.insert_rows()`或直接赋值方式插入行后，以下三类公式不会自动更新：
1. **SUM范围**：合计行`SUM(F6:F28)`不会自动扩展为`SUM(F6:F38)`
2. **合计行引用**：如`I29`不会自动变为`I42`
3. **跨sheet引用**：汇总表中`='3-5应收账款'!I29`不会自动变为`='3-5应收账款'!I42`

**影响**：合计数据错误 → 汇总表数据错误 → 勾稽失败

**预防**：插入行后立即执行公式修复脚本（Phase 3.1），逐项检查三类公式

**受影响Sheet**：3-5应收账款、5-6预收款项、5-10-3其他应付款、5-5应付账款

## 教训2：插入行格式丢失（反复出现）

**问题**：Python脚本插入行后，新行继承默认格式（字号12pt、对齐None、无边框、数字格式General），与模板行格式（11pt、按列设置对齐、有边框、会计格式`#,##0.00`）不一致

**影响**：打印/显示时格式不统一，不专业，需返工

**预防**：插入行后立即执行格式复制脚本（Phase 3.2），逐列从参考行复制font/alignment/border/number_format/row_height

**特别注意**：E列(币种)字体必须为"宋体"而非Times New Roman

## 教训11：openpyxl保存含公式Excel时Excel弹窗要求保存

**问题**：openpyxl写入公式时只存公式文本不存缓存值。Excel/WPS打开文件后自动重算公式，检测到"文件有变化"，弹窗"是否保存更改"。每次打开都会弹，非常烦人。

**影响**：用户每次打开文件都需要手动确认保存，影响使用体验

**预防**：在`wb.save()`前添加：
```python
from openpyxl.workbook.properties import CalcProperties
wb.calculation = CalcProperties(calcId=0, fullCalcOnLoad=0)
```
- `calcId=0`：标记文件已计算过
- `fullCalcOnLoad=0`：打开时不需要重新计算

**⚠️ 重要更新（2026-05）**：上述CalcProperties方案**无法根治弹窗**。openpyxl保存时会清除公式缓存值（`<v>`元素），WPS/Excel打开后重算发现文件"有变化"仍弹"是否保存更改"。**正确方案：openpyxl保存后用Excel COM重算保存**：
```python
import win32com.client, time
app = win32com.client.Dispatch('Excel.Application')
app.Visible = False; app.DisplayAlerts = False
wb = app.Workbooks.Open(filepath)
app.Calculate()
time.sleep(2)
wb.Save()
wb.Close(SaveChanges=False)
app.Quit()
```

这样Excel会把公式缓存值写回文件，WPS再打开时不会重算也就不会弹窗。

**适用范围**：所有通过openpyxl生成/修改的Excel文件（评估明细表、待补资料清单、成本法底稿等）

## 教训12：坏账准备与预计风险损失的评估处理

**问题**：评估结论中没有"坏账准备"概念，对应概念是"预计风险损失"。坏账准备的评估值应为0，预计风险损失的金额应等于坏账准备金额。

**正确处理**（v3.26更正）：
- "减：坏账准备"行(A列="坏账准备")：I列(账面价值)=减值/跌价准备金额，J列(评估价值)=0
- "减：预计风险"行(A列="预计风险")：I列(账面价值)=0，J列(评估价值)=减值/跌价准备对应评估值
- 最终合计公式：J = 合计1的J - 坏账准备的J(0) - 预计风险的J

**影响范围**：3-5应收账款、3-8-3其他应收款等含坏账准备的科目

## 教训15：汇总表隐藏行不能依赖data_only模式判断

**问题**：openpyxl的data_only=True模式读取公式单元格返回None（如果文件未在Excel中保存过），导致有数据的汇总行被误判为零而隐藏。

**预防**：隐藏汇总表行时，应基于已知有数据的科目清单判断，而非仅依赖data_only读取的D/E列值。data_only模式下None≠0，不应将None视为零值。

## 教训16：隐藏的是无数据工作表，不是零值行

**问题**：用户要求"隐藏账面值和评估值均为0的科目"，实际意图是隐藏**整个工作表**（sheet），而非隐藏行。之前错误地隐藏了1792行数据行，导致有数据的行也被隐藏。

**影响**：数据行被隐藏后无法正常查看，且汇总表中data_only=True读取公式返回None导致有数据行也被误隐藏。

**正确做法**：
1. **取消所有行隐藏**：先恢复所有行可见状态
2. **识别无数据工作表**：通过动态查找"账面价值"和"评估价值"列标题，检查数据行是否有非零数值（注意不同工作表的值列位置不同：E/F、G/J、D/E、H/I等）
3. **隐藏工作表**：将无数据的工作表设置`ws.sheet_state = 'hidden'`
4. **保留有数据的汇总表**：汇总表虽无硬编码数值但有公式引用子表，需根据子表数据判断是否保留可见
5. **系统辅助工作表默认隐藏**：设置、0-其他方法结论、设定信息虽为结构性工作表，但非评估数据展示用途，默认隐藏
6. **隐藏条件**：账面值**且**评估值**同时**为0才隐藏，仅一项为0不隐藏（DT-20明确化）
7. **集中执行**：所有隐藏操作MUST在Phase 5交付前集中执行（DT-110），禁止Phase 2-4期间分散执行

**关键代码**：
```python
# 动态查找值列位置
for row in ws.iter_rows(min_row=1, max_row=10):
    for cell in row:
        if cell.value == '账面价值':
            book_col = cell.column
        elif cell.value == '评估价值':
            eval_col = cell.column

# 隐藏工作表而非行
ws.sheet_state = 'hidden'
```

## 教训19：系统辅助工作表默认隐藏（DT-23）+ 隐藏操作集中执行（DT-110）

**问题**：设置、0-其他方法结论、设定信息等工作表是模板内部系统表，用于存储公式参数和模板配置，非评估数据展示用途，对用户可见会干扰界面。河南平绿项目交付时这3个系统辅助表+1个无数据表(00000000)均未按规则隐藏，根因=隐藏操作分散在S5的两个步骤中且与COM重算/页脚设置混排，容易被遗漏。

**正确做法**：
1. 系统辅助工作表（设置、0-其他方法结论、设定信息）默认隐藏（`ws.sheet_state='hidden'`）
2. 设定信息sheet**不属于"直接跳过隐藏"**——MUST在隐藏前先填写B6=被评估单位全称、B7=评估基准日
3. **所有sheet隐藏操作MUST在Phase 5交付前集中执行（DT-110）**，禁止Phase 2-4期间分散执行
4. 隐藏无数据工作表时，账面值**且**评估值**同时**为0才隐藏，仅一项为0不隐藏
5. 执行顺序：填写设定信息→扫描无数据表→隐藏系统辅助表→确认辅汇总表→输出隐藏清单→保存

## 教训20：删除行后公式引用旧行号导致数据错误

**问题**：电子设备表删除行33-38（6行空行）后，合计行从原行39移至行33，但行35（最终合计）的公式仍引用旧行号`=G39-G40`，应更新为`=G33-G34`。同时4-8固定资产汇总表引用电子设备的行号`J39`/`K39`也未同步更新。

**影响**：公式引用不存在的行号产生#REF!错误或数据错乱

**预防**：删除行后必须执行与插入行相同的三类公式修复：
1. SUM范围收缩（如`SUM(J7:J32)`不变则无需修改）
2. 行内自引用行号更新（如`G39-G40`→`G33-G34`）
3. 跨sheet引用行号更新（汇总表中`J39`→`J33`）
4. 清理残留合并单元格（行号超出实际行数的A:C合并需unmerge）

**关键代码**：
```python
# 删除行后修复行内自引用
for c in range(1, max_col+1):
    cell = ws.cell(row=new_row, column=c)
    if isinstance(cell.value, str) and cell.value.startswith('='):
        cell.value = cell.value.replace(str(old_row), str(new_row))

# 清理残留合并
for m in list(ws.merged_cells.ranges):
    if m.min_row > ws.max_row:
        ws.unmerge_cells(str(m))
```

**问题**：电子设备表行33-38为空行，之前仅清除了边框（`Border()`），但空行仍存在，与上下文格式不一致（数据行有边框，空行无边框，混合显示不专业）。用户要求"要么删除，要么全部保留框线"。

## 教训22：打印范围必须精确匹配实际内容

**问题**：3-7预付款项print_area为`$A$1:$M$86`，但实际数据只到行28，行29-86全为空白，导致打印时出现大量空白页。

**影响**：打印浪费纸张，且不专业

**正确做法**：每个工作表的print_area必须精确覆盖至最后有效行（含合计行），不超出。删除行或隐藏行后必须同步调整。如3-7预付款项应设为`$A$1:$M$28`。

**检查方法**：
```python
# 找到最后有内容的行
last_data_row = 0
for r in range(1, ws.max_row + 1):
    for c in range(1, ws.max_column + 1):
        if ws.cell(row=r, column=c).value is not None:
            last_data_row = r
ws.print_area = f'A1:{openpyxl.utils.get_column_letter(ws.max_column)}{last_data_row}'
```

## 教训23：明细表页脚必须设置实际人员信息

**问题**：评估明细表模板中页脚为占位符（"被评估单位填表人：张三"、"评估人员：张三 李四"），未替换为实际人员。且汇总表和明细表的页脚设置不统一——部分明细表有页脚、部分为空。

**正确做法**：
- **明细表**（标题不含"汇总"）：设置oddFooter和evenFooter
  - 左侧：`被评估单位填表人：{姓名}\n填表日期：{YYYY年MM月DD日}`
  - 右侧：`评估人员：{姓名1}  {姓名2}`
- **汇总表**（标题含"汇总"）：不设页脚
- **系统表**（00000000等）：不设页脚

**关键代码**：
```python
footer_left = '被评估单位填表人：吴一凡\n填表日期：2026年05月16日'
footer_right = '评估人员：史久霞  黄伟'

for ws in wb.worksheets:
    if ws.sheet_state == 'hidden':
        continue
    is_summary = '汇总' in ws.title or ws.title == '00000000'
    if not is_summary:
        ws.oddFooter.left.text = footer_left
        ws.oddFooter.right.text = footer_right
        ws.evenFooter.left.text = footer_left
        ws.evenFooter.right.text = footer_right
```

## 教训24：序时账科目编码与评估明细表编码体系不同

**问题**：填写发生日期时，直接在序时账中按科目余额表编码搜索，发现部分科目无匹配。例如：
- 其他流动资产（科目余额表编码1234）在序时账中无直接条目，需通过审计调整分录中"进项"关键词间接查找（对应"待抵扣增值税进项税额"）
- 合同负债（科目余额表编码2131）在序时账中无直接条目，需通过基准日报表中的合同签订时间确定
- 递延所得税资产（序时账编码1811）虽有序时账记录，但编码与科目余额表可能不完全对应

**影响**：按错误编码搜索导致"找不到数据"，误判科目无发生记录

**预防**：
- 序时账的科目编码体系与科目余额表/评估明细表**不同**，不可直接用科目余额表编码去序时账中搜索
- 找不到匹配时，应扩大搜索策略：
  1. 先在序时账中按科目名称关键词模糊搜索（如"进项"、"递延"）
  2. 查看审计调整分录/调整分录工作表，找到对应科目的辅助信息
  3. 查看基准日报表中的合同/项目信息
  4. 交叉比对科目余额表的科目名称与序时账的科目名称
- 关键编码映射示例（绿城熵里项目）：

| 科目余额表编码 | 科目名称 | 序时账中查找方式 |
|--------------|---------|----------------|
| 1234 | 其他流动资产 | 审计调整分录中"进项"→待抵扣增值税进项税额 |
| 1811 | 递延所得税资产 | 序时账code 1811直接搜索 |
| 2131 | 合同负债 | 基准日报表LCJW24034合同签订时间 |
| 2221xxx | 应交税费（增值税） | 序时账中按2221前缀+明细搜索 |

## 教训25：COM resave会破坏datetime类型单元格

**问题**：使用openpyxl写入datetime对象后，调用Excel COM重算保存（`DispatchEx` → Open → Calculate → Save → Close），COM会将datetime单元格转换为int序列号（Excel serial number），同时覆盖number_format。

**影响**：发生日期列从"2023年1月"变为44927（Excel序列号），日期格式完全丢失

**正确流程**：
```
openpyxl写入datetime → wb.save() → COM resave → openpyxl二次修复datetime
```

**关键代码**：
```python
import openpyxl
from datetime import datetime

# 第一步：openpyxl写入datetime
ws['C6'] = datetime(2023, 1, 5)
ws['C6'].number_format = 'yyyy"年"m"月"'
wb.save(filepath)

# 第二步：COM resave（会破坏datetime）
resave_with_cached_values(filepath)  # 调用附录C的COM重算函数

# 第三步：openpyxl二次修复datetime
wb2 = openpyxl.load_workbook(filepath)
ws2 = wb2['目标Sheet']
# COM将datetime转成int，需重新写入
ws2['C6'] = datetime(2023, 1, 5)
ws2['C6'].number_format = 'yyyy"年"m"月"'
wb2.save(filepath)
# 注意：二次修复后不再做COM resave，否则又会破坏datetime
```

**预防**：
- 涉及datetime的修改后，COM resave必须紧跟openpyxl二次修复
- 二次修复后**不再做COM resave**，避免循环破坏
- 如果仅有datetime修改（无公式变更），可以只做openpyxl保存+一次COM resave+openpyxl修复
- 如果无datetime单元格被修改，COM resave后无需二次修复

## 教训26：行操作后标题行合并单元格丢失

**问题**：对明细表执行行恢复/插入行/删除行操作后，标题行（通常是第1-2行）的合并单元格可能丢失。例如5-8职工薪酬的A1:H1（标题合并）和A2:H2（日期合并）在行恢复操作后丢失，导致标题和日期行显示错乱——文字只显示在A列，B-H列为空。

**影响**：标题行显示不完整，格式不专业

**预防**：
- 任何行操作（插入/删除/恢复）后，**必须检查标题行合并单元格是否完整**
- 标准检查流程：
  ```python
  # 检查并恢复标题行合并
  expected_merges = ['A1:H1', 'A2:H2']  # 大部分明细表的标准标题合并
  current_merges = [str(m) for m in ws.merged_cells.ranges]

  for merge in expected_merges:
      if merge not in current_merges:
          ws.merge_cells(merge)
          # 设置居中对齐
          ws[merge.split(':')[0]].alignment = Alignment(horizontal='center', vertical='center')
  ```
- 不同科目类型的标题合并可能不同（如固定资产子表可能有A1:I1），需参考同类型科目模板确认

## 教训27：多余小计行导致汇总值翻倍

**问题**：5-8职工薪酬仅1条数据行（行6），但行7出现了A7="合计"、D7=ROUND(SUM(D6:D6),2)的小计行。同时行27存在正式的合计行（SUM(D6:D26)），行7的值被行27的SUM范围再次包含，导致D6值被计算两次——汇总值=实际值的2倍。

**根因**：删行/格式修复过程中，中间小计行未被正确清理。行7本应是空数据行（预留行），但残留了"合计"文字和SUM公式。

**影响**：该科目账面价值、评估价值均翻倍，导致汇总表和分类汇总表数据错误

**预防**：
- 删行/格式修复后，必须逐行检查数据区域（第1条数据行到合计行之间）：
  1. 不应有多个"合计"行——每个明细表只有1个合计行（通常在最后一行或倒数第二行）
  2. 空数据行应清空所有内容（A列序号、D列业务内容、公式列等），不是保留SUM公式
  3. 检查合计行SUM范围是否包含不应有的中间汇总行
- 标准空数据行格式：
  ```python
  # 空数据行：仅保留E列"人民币"、公式列（H=I-D、I=IF...），其余清空
  ws.cell(row=r, column=1).value = None  # A列清空
  ws.cell(row=r, column=2).value = None  # B列清空
  ws.cell(row=r, column=3).value = None  # C列清空
  ws.cell(row=r, column=4).value = None  # D列清空
  # E列保留"人民币"（如模板原有）
  ws.cell(row=r, column=5).value = '人民币'
  # F-H列：保留公式或清空
  ws.cell(row=r, column=8).value = None  # H列清空（非合计行不应有数值）
  ```
- **合计行唯一性验证**：每个明细表grep "合计"，应只有1个匹配（除标题行外）

## 教训28：打印区域列范围超出数据列，右侧空白打印区域

**问题**：3-7预付款项的print_area设置为`$A$1:$S$28`，但实际数据只到M列（13列），N-S列（14-19列）无任何数据。这导致打印预览时右侧出现大片空白区域，打印输出右侧空白。

**根因**：前次修复时将print_area从`$A$1:$M$86`调整为`$A$1:$M$28`，但后来又有操作将列范围扩展到S列。openpyxl的max_column可能因残留格式/边框而返回超出实际数据列的值。

**影响**：打印预览右侧空白，打印输出不美观

**预防**：
- 设置print_area时，**不仅要检查行范围，还要检查列范围**
- 列范围应精确匹配实际数据列，不可使用max_column作为依据（max_column可能包含无数据但有格式的列）
- 标准检查流程：
  ```python
  # 确定实际数据列范围
  actual_max_col = 0
  for col in range(1, ws.max_column + 1):
      has_data = any(ws.cell(row=r, column=col).value is not None for r in range(1, ws.max_row + 1))
      if has_data:
          actual_max_col = col

  # 设置print_area时使用actual_max_col
  from openpyxl.utils import get_column_letter
  col_letter = get_column_letter(actual_max_col)
  ws.print_area = f"'{ws.title}'!$A$1:${col_letter}${ws.max_row}"
  ```
- 对于3-7预付款项这类科目：数据列A-M（序号~备注），print_area列范围应为$M
- 对于3-10合同资产这类科目：数据列A-I（序号~备注），print_area列范围应为$I

## 教训29：小计/减值行紧贴数据行，中间空白行无格式

**问题**：3-10合同资产中，12项数据行（6-17行）后面紧跟3行非数据行（18=小计，19=减值准备，20=减值损失），然后是7行完全空白无格式的行（21-27），最后是合计行（28）。打印时，小计/减值行紧贴数据行，与数据行之间无间隔，而合计行前又有大片无格式的空白区域。

**正确布局**：数据行 → 空白行（有格式）→ 小计/减值行 → 合计行

- 行18-24：7行空白行（thin边框+row_height=16.5+居中对齐，与数据区格式一致）

- 中间空白行必须补齐格式：thin边框、row_height=16.5、居中对齐、字体Times New Roman 11pt
- 行内容移动的标准流程：
  1. 保存原始行的值和格式
  2. 取消原始行的合并单元格
  3. 清空原始行，应用数据区标准格式
  4. 写入新行，恢复合并单元格（A:C合并）
  5. 更新本表公式引用
  6. 更新跨sheet引用

## 教训31：空白无格式行穿插在明细表数据行与合计行之间

**问题**：5-8职工薪酬中，行26（最后数据行）和行28（合计行）之间插入了行27——完全空白、无边框、无对齐、无数据。类似问题此前在4-8-6电子设备中也出现过（行33/34与行36之间有空白行35）。这是模板残留或行操作失误导致的共性问题。

**特征识别**：空白行位于数据行与合计行之间，且：
- 无任何值（所有列value=None）
- 无边框（border.left/right/top/bottom.style=None）
- 无对齐（alignment.horizontal/vertical=None）
- 有row_height=16.5（继承自模板行高设置）

**影响**：格式不一致（有格式的数据行紧邻无格式的空白行），打印时不美观，且反复出现同类问题

**修复流程**：
1. 保存合计行的值和格式
2. 取消合计行和空白行的合并单元格
3. 清空空白行和原合计行
4. 将合计行内容写入空白行位置，更新公式中的自引用行号（如=E28-D28→=E27-D27）
5. 恢复合并单元格（A:C合并）
6. 更新print_area
7. 全局搜索汇总表中对旧行号的引用，更新为新行号

**具体案例**：
- 5-8职工薪酬：删除行27（空白），行28合计行上移至行27，5-流动负债汇总!D13/E13从引用D28/E28改为D27/E27
- 4-8-6电子设备：删除行35（空白），行36合计行上移至行35，自引用公式从36改为35

**预防**：
- 明细表中数据行与合计行之间不允许出现空白无格式的行——T42纪律规则
- 填写完成后必须检查：逐行扫描数据行到合计行之间是否存在无格式空白行
- 发现即删除，将合计行上移，同步更新公式、合并单元格、打印区域、跨表引用

- 完成评估明细表交付时，主动询问用户是否需要检查

- 🔴极端：破产重整（资产<负债）、失信+母公司破产等

4. **多关键词OR逻辑**：一家公司可用多个关键词，任一匹配即归入

## 教训37：泛匹配项用全部科目末笔日期更新=日期不具代表性

**问题**：3项核实规则中keywords=None（"其他"/"个人(押金)"/"河北格乐熙"），脚本用全部科目记录的末笔日期作为核实结果。但：
- "其他"（其他应收款）包含多个不同业务（备用金、押金、暂付款等），末笔日期无法代表特定业务
- "个人(押金)"（其他应付款）在序时账摘要中搜索不到"押金"关键词
- "河北格乐熙"在序时账中搜索不到任何"格乐熙"匹配，可能入账名称完全不同

**根因**：泛匹配项（keywords=None）意味着无法精确识别特定客商/业务的序时账记录，使用全部科目末笔日期等价于"用整个科目最晚一笔的日期"，对特定结算对象不具代表性。

1. keywords=None的条目**禁止**用全部科目末笔日期更新
2. 脚本中keywords=None时直接跳过，输出警告
3. 保留原始日期不变
4. 在核实结果汇总中列为"无法自动核实"项，提示人工核实
5. 具体公司名在序时账中搜不到匹配时同样跳过并标注原因

## 教训41：隐藏汇总表引用未同步更新导致勾稽失败

**问题**：`3-辅-流动资产汇总`、`3-9-辅-存货汇总`、`8-减值准备汇总表`、`9-非财务信息汇总表`四张表默认为隐藏状态。修改明细表（插入行/删除行/行内容移动）后，仅更新了可见汇总表（如`3-流动资产汇总`、`2-分类汇总`）的跨sheet引用行号，遗漏了这些隐藏汇总表的引用更新。导致隐藏汇总表中公式引用旧行号→指向空单元格或错误行→汇总数据错误。Phase 3.5自动化验证检查4中原有`if ws_sum.sheet_state == 'hidden': continue`逻辑跳过了隐藏汇总表，使错误无法被自动检测到，直到取消隐藏或最终复核时才暴露。

**根因**：
1. 隐藏状态给操作者"不存在"的心理暗示，修复跨sheet引用时仅关注可见sheet
2. Phase 3.5自动化验证中检查4主动跳过hidden汇总表，等于豁免了隐藏表的验证义务
3. T43虽提到"辅汇总表和减值准备汇总表可能引用小计行或减值准备行"，但未明确这些表的默认隐藏属性及必须同等对待
4. 评估明细表模板中这些表始终默认隐藏，操作者容易忽略
5. `9-非财务信息汇总表`引用了固定资产/无形资产子表的合计行，子表插入数据行后合计行下移，引用行号未同步更新

**修复**：
1. 新增T61执行纪律——四个隐藏汇总表必须与可见汇总表同等对待，修改行号时必须同步更新其引用
2. Phase 3.5自动化验证检查4中删除`if ws_sum.sheet_state == 'hidden': continue`逻辑
3. Phase 3自检清单增加"3-辅-流动资产汇总/3-9-辅-存货汇总/8-减值准备汇总表/9-非财务信息汇总表引用已同步更新（T61）"检查项
4. Step 1.1模板Sheet结构列表中标注四个隐藏汇总表及T61引用

**预防**（T61）：隐藏汇总表同步更新规则：
1. `3-辅-流动资产汇总`、`3-9-辅-存货汇总`、`8-减值准备汇总表`、`9-非财务信息汇总表`始终默认隐藏（`sheet_state='hidden'`），不论子表是否有数据
2. 隐藏≠不更新——修改明细表行号变动时，必须遍历**所有**汇总表（含hidden状态）更新引用
3. Phase 3.5自动化验证不跳过hidden汇总表

## 教训42：辅汇总表与可见汇总表引用逻辑不同，简单复制可见表公式导致辅表数据错误

**问题**：修复`3-辅-流动资产汇总`链接时，将可见表`3-流动资产汇总`的公式直接复制到辅表，导致：
1. 辅表D列（账面价值）从引用小计行(毛额)变为引用合计行(净额)——辅表D列展示的数据从"账面毛额"变成了"账面净额"，与可见表雷同，失去辅助验证意义
2. 辅表E列引用了I列(账面价值)而非J列(评估价值)——列引用错误
3. `8-减值准备汇总表`引用了合计行而非减值准备行——减值金额取到净额而非减值额

**根因**：
1. 辅表与可见表的设计意图不同：辅表D列专用于展示"账面毛额"（未扣减值的原始金额），以便与E列"评估净额"对比验证；可见表D/E列均展示净额
2. 修复时凭直觉"隐藏表=可见表的副本"，未对照原始模板逐一确认引用目标
3. 减值准备汇总表的引用目标是"减：坏账准备"行，而非小计行或合计行，这一点在T43中提及但未形成明确规则

**具体修复案例**（绿城熵里项目v3.0）：

| 表 | 位置 | 修复前（错误） | 修复后（正确） | 错误原因 |
|----|------|-------------|-------------|---------|
| 3-辅-流动资产汇总 | E列各科目行 | 引用小计行(如I26) | 引用合计行(如I28/I29) | E列应引用净额但引用了毛额 |
| 3-辅-流动资产汇总 | D10(其他应收款) | `='3-8-3其他应收款'!I27` | `='3-8-3其他应收款'!I24` | D列应引用小计行(毛额)但引用了合计行(净额) |
| 3-辅-流动资产汇总 | E7(其他应收款) | `='3-8-3其他应收款'!I27` | `='3-8-3其他应收款'!J27` | E列应引用J列(评估价值)但引用了I列(账面价值) |
| 8-减值准备汇总表 | Row8(应收账款) | I27/J27(合计行) | I25/J25(减值行) | 减值表应引用减值准备行而非合计行 |

| 行类型 | 含义 | I列(账面价值) | J列(评估价值) |
|--------|------|-------------|-------------|
| 数据行 | 各结算对象 | 各户金额 | 各户金额 |
| **小计行** | 毛额（未扣减值） | 毛额合计 | 毛额合计 |
| **减：减值准备行** | 坏账准备/预计风险损失 | 减值金额 | 0（评估值=0） |

|----|-------------|-------------|-------|

| 3-流动资产汇总(可见) | **合计行**(净额) | **合计行**(净额) | — |

**修复**：
1. 新增T62执行纪律——辅汇总表与可见汇总表引用逻辑不同，修复时禁止简单复制可见表公式

## 教训44：openpyxl缺少Excel事件引擎——裸insert_rows/delete_rows导致6类问题

**问题**：Agent在明细表/底稿中使用`ws.insert_rows()`插入行后，产生6类连锁问题：
1. **SUM范围断裂**：合计行`SUM(F6:F28)`不会自动扩展为`SUM(F6:F38)`（教训1）
2. **格式丢失**：新行为裸行——12pt/无边框/General vs 模板行11pt/有边框/#,##0.00（教训2）
3. **合并单元格错位**：插入行后A:C列可能被错误合并（教训13），合计行合并范围丢失（教训18）
4. **打印范围未扩展**：新增行在打印范围外（教训14）
5. **跨sheet引用过期**：汇总表`='3-5应收账款'!J29`不随行号变化更新（教训20）
6. **标题行合并丢失**：多行表头合并单元格被破坏（教训26）

**根因**：Excel GUI插入行时触发完整事件链（格式继承→SUM延伸→引用更新→合并修复→打印范围），而openpyxl的`insert_rows()`仅执行XML行下移，不触发任何回调。Agent缺少Excel的"事件引擎"。

**修复**：封装`excel_row_ops.smart_insert_row()`（位于`valuation-common/scripts/excel_row_ops.py`），一次调用完成：
- `ws.insert_rows()` + 格式继承（copy参考行font/alignment/border/number_format/height）
- SUM范围自动扩展（合计行公式重写）
- 合并单元格修复（取消数据行错误合并，保留合计行合并）
- 打印范围更新
- 跨sheet引用行号同步

**规则**：
- DT-113：禁止裸`ws.insert_rows()`/`ws.delete_rows()`，MUST使用smart_insert_row/smart_delete_rows
- DT-2：插入行后修复三类公式 → smart_insert_row自动完成
- DT-112：最小格式继承 → smart_insert_row自动完成

**使用示例**：
```python
from excel_row_ops import smart_insert_row, smart_insert_rows_for_data

# 方式1：手动指定位置插入
result = smart_insert_row(ws, target_row=25, ref_row=24, count=3,
                          total_row=25, wb=wb)

# 方式2：按数据量自动插入（推荐）
result = smart_insert_rows_for_data(ws, data_count=10, wb=wb)
```

## 教训46：多脚本分批写入导致数据丢失/覆盖（河南平绿2025复盘，2026-05-23）

**问题**：Phase 2填写过程使用了多个Python脚本分批执行（fill_phase2.py → fill_execute_v3.py → fill_patch.py → 固定资产/存货脚本），导致：

**预防**：
- **DT-124**：Phase 2所有sheet数据填写MUST在单个Python脚本中完成，末尾只调用一次wb.save()
- 脚本必须包含try/except，异常时输出已成功写入的sheet清单
- 如果脚本确实过长，采用"子函数模式"（一个脚本内定义各sheet填写函数，主函数依次调用，最后统一save），而非"子脚本模式"

## 教训49：Python f-string反斜杠语法错误（河南平绿2025复盘，2026-05-23）

**问题**：Python脚本中使用`f'{" | ".join(vals)}'`格式化输出时，触发`SyntaxError: f-string expression part cannot include a backslash`。

**根因**：Python 3.12中f-string的大括号表达式内不允许包含反斜杠（`\`），包括转义字符如`\n`、`\\`等。`" | ".join(vals)`中的`\"`被解析为f-string内的反斜杠转义。

- **DT-127**：f-string中需要换行/特殊字符时，先赋值给变量再引用
- 正确写法：`sep = " | "; print(f'{sep.join(vals)}')`
- 错误写法：`f'{" | ".join(vals)}'`

## 教训54：手动处理评估明细表必须先全量扫描表头结构（卫导项目2025复盘，2026-05-25）

**问题**：卫导项目手动处理明细表时，Agent未先全量扫描所有sheet的Row5表头结构就编写填写脚本，导致：
1. 不知道5-7合同负债用"结算内容"而非"业务内容"→53行漏填
2. 不知道5-5应付账款的列序（发生日期col3+业务内容col4）与3-5应收账款（业务内容col3+发生日期col4）相反→填错列风险（DT-46）
3. 不知道哪些sheet有"内容"类列、哪些没有→遗漏非往来科目

**根因**：

1. **dt_runner流程的自动列映射被绕过**：dt_runner在Phase 1生成sheet_col_map.json时已全量扫描表头并记录每个sheet的列位映射。手动处理跳过了这一步，Agent凭"上次见过"的印象假设列名和列序。

2. **"先写再验"而非"先扫再写"**：Agent直接编写写入脚本，而非先执行一遍只读扫描（读取Row5表头→输出列名清单→确认列位→再编写写入脚本）。

**标准"先扫再写"流程**（手动处理时必须执行）：

```python
# Step 1: 全量扫描表头结构（只读，不写入）
from openpyxl import load_workbook
wb = load_workbook(filepath, data_only=True)
for sname in wb.sheetnames:
    ws = wb[sname]
    headers = {}
    for cell in ws[5]:  # Row 5 = 标准表头行
        if cell.value is not None:
            headers[cell.column] = str(cell.value).strip()
    # 检查是否存在"内容"类列
    content_cols = [(c, n) for c, n in headers.items() 
                    if any(kw in n for kw in ['业务内容', '结算内容', '款项内容', '核算内容'])]
    # 检查是否存在"日期"类列
    date_cols = [(c, n) for c, n in headers.items() if '发生日期' in n]
    if content_cols or date_cols:
        print(f"{sname}: 内容列={content_cols}, 日期列={date_cols}")
```

```python
# Step 2: 确认列位映射后再编写写入逻辑
# 关键：不同sheet的"内容"列和"日期"列的列序可能相反（DT-46）
# 资产类：内容在前(左)，日期在后(右)
# 负债类：日期在前(左)，内容在后(右)
```

**预防**：

- **DT-180（建议新增）**：手动处理评估明细表时，编写写入脚本前MUST先执行全量表头扫描（只读），输出所有sheet的列名→列号映射清单，确认"内容"列和"日期"列的列位后再编写写入逻辑。禁止跳过扫描步骤直接假设列名/列序。
- **通用原则**：对任何结构不完全确定的Excel模板，写入前必须先只读扫描表头，确认列名和列序。"上次见过"≠"这次一样"。

## 教训55：汇总行序号非数字——跳过汇总行的判断标准（卫导项目2025复盘，2026-05-25）

**问题**：卫导项目中，多个sheet的"业务内容"列有少量空白行（4行左右），这些行的序号列内容为"合　计""减：坏账准备""减：预计损失""合　计"，属于汇总/减值行而非数据行，本身不需要填写业务内容。但脚本未区分数据行和汇总行，将它们计入"未填写"数量，误导了对填写完整性的判断。

**根因**：

1. **数据行判断条件不精确**：脚本用`seq is not None and seq != ''`判断数据行，但汇总行的序号也满足此条件（"合　计"非空非None）。正确判断应为`isinstance(seq, (int, float))`或`seq不是文字`。

2. **汇总行的标准特征**：
   - 序号列内容为文字（"合　计""减：坏账准备""减：预计损失""合　计"）
   - 名称列（B/C列）通常为空
   - 不需要填写业务内容/发生日期

3. **A列标记体系已解决此问题**：模板v1.90起A列有`合计1`/`坏账准备`/`预计风险`/`合计2`标记，dt_runner据此区分行类型。但手动处理时未使用此标记体系。

**汇总行标准结构**（以3-5应收账款为例）：

| 行 | A列标记 | 序号列 | 名称列 | 是否填内容 |
|----|--------|-------|--------|----------|
| 数据行 | (空) | 1, 2, 3... | 结算对象名称 | ✅ 需填 |
| 合计1行 | `合计1` | "合　计" | (空) | ❌ 不填 |
| 坏账准备行 | `坏账准备` | "减：坏账准备" | (空) | ❌ 不填 |
| 预计风险行 | `预计风险` | "减：预计损失" | (空) | ❌ 不填 |
| 合计2行 | `合计2` | "合　计" | (空) | ❌ 不填 |

**预防**：

- 手动处理时，数据行判断条件应改为：`isinstance(seq, (int, float)) and seq > 0`
- 或使用A列标记：`ws.cell(row=r, column=1).value not in [None, '合计1', '坏账准备', '预计风险', '合计2']`
- 填写完整性统计应排除汇总行，只统计数据行的填充率

## 教训56：新准则调入科目序时账中无独立记录——合同资产/合同负债须溯源调整前科目（卫导项目2025复盘，2026-05-25）

**问题**：合同资产（3-10合同资产67行）和合同负债（5-7合同负债52行）在序时账中无任何独立记录（搜"合同资产"=0条，搜"合同负债"=0条），导致Phase 3核实发生日期时全部匹配失败。其他非流动资产（4-18其他非流动资产24行）同理在序时账中0条独立记录。

**根因**：

1. **新会计准则的重分类机制**：根据《企业会计准则第14号—收入》（2017年修订，即新收入准则），合同资产和合同负债是从应收账款和预收账款中重分类出来的科目：
   - **合同资产**：企业已向客户转让商品而有权收取对价的权利，且该权利取决于时间流逝之外的其他因素（如质保金、保留金）。在旧准则下这部分作为应收账款列示，新准则将其单独列示为合同资产。序时账中仍以"应收账款"科目入账。
   - **合同负债**：企业已收或应收客户对价而应向客户转让商品的义务。在旧准则下作为预收账款列示，新准则将其单独列示为合同负债。序时账中仍以"预收账款"科目入账。
   - **其他非流动资产**：部分往来从应收账款等科目重分类而来（每个项目来源不同），序时账中无独立科目。

2. **会计软件的实际处理**：大多数财务软件（金蝶/用友等）在序时账中仍以调整前的科目编码入账，仅在出具报表时按新准则重分类列示。因此序时账中搜索"合同资产"/"合同负债"=0条记录。

3. **通用映射规则（默认值，项目特殊时需覆盖）**：

| 调入科目 | 调整前原始科目 | 会计准则依据 | 方向规则 |
|---------|-------------|------------|---------|
| 合同资产 | 应收账款 | CAS14-2017：无条件收款权→应收账款，附条件收款权→合同资产 | 借方（资产类） |
| 合同负债 | 预收账款 | CAS14-2017：履约义务相关预收款→合同负债，非履约相关→预收账款 | 贷方（负债类） |
| 其他非流动资产 | 视项目来源（常见：应收账款） | 重分类调整，无统一准则 | 借方（资产类） |

**卫导项目具体修复**：

| 科目 | 溯源科目 | 方向 | 修复行数 | 典型变化 |
|------|---------|------|---------|---------|
| 合同资产(3-10) | 应收账款 | 最近借方 | 67行 | 湖南中电星河 2023-06→2025-12-31 |
| 合同负债(5-7) | 预收账款 | 最近贷方 | 52行 | 整体从2022-2023年→2025年 |
| 其他非流动资产(4-18) | 应收账款 | 最近借方 | 24行 | 全部匹配成功 |

**预防**（DT-181）：

1. **Phase 0确认映射**：Phase 0 Step 0.5数据提取时，MUST确认合同资产/合同负债/其他非流动资产的序时账溯源科目。默认映射=上表，项目特殊时覆盖。
2. **Phase 3自动溯源**：journal_extractor.py在搜不到调入科目的序时账记录时，MUST自动尝试默认溯源科目，而非直接返回0匹配。
3. **方向规则不因溯源改变**：合同资产溯源于应收账款，仍按资产类规则取最近借方日期；合同负债溯源于预收账款，仍按负债类规则取最近贷方日期。
4. **项目来源差异**：其他非流动资产的来源科目不统一（本项目=应收账款，其他项目可能是其他应收款或长期应收款），MUST在Phase 0确认。

**受影响项目**：湖南卫导信息科技有限公司2025评估明细表

## 教训57：插入行后格式破坏——裸insert_rows/delete_rows绕过格式管线（邦能达项目2026-05复盘）

**问题**：通过裸 `ws.delete_rows()` + `ws.insert_rows()` 操作固定资产明细表的数据行后，出现以下问题：
1. **字体错误**：数据行字体变为 Times New Roman/12pt（应为 宋体/TNR 11pt）
2. **边框丢失**：thin 边框丢失
3. **数字格式丢失**：金额列失去 `#,##0.00` 格式
4. **序号错位**：删除+插入行后序号未重新编号
5. **尾部多余空行**：`max_row` 未随 delete_rows 缩减，残留空行
6. **合计行 B:D 合并单元格丢失**：删除数据行时合计行的合并范围被破坏

**根因**：

1. **DT-165 格式管线被绕过**：Skill 规定 MUST 通过 `fill_sheet()` 操作数据区域。`fill_sheet()` 内部调用 `smart_insert_row()` 进行插行（保留 SUM 公式/合并单元格/条件格式），并在写入后调用 `_apply_direct_format()` 应用标准格式（宋体/TNR 11pt, thin 边框, #,##0.00）。

2. **裸 insert_rows 不修复任何东西**：`ws.insert_rows()` 仅创建空行，不复制格式/公式/合并单元格。新行继承 Excel 默认格式（等线/11pt/无边框），与模板要求不一致。

3. **裸 delete_rows 破坏合并单元格**：`ws.delete_rows()` 会连带删除合并单元格范围，但合计行的 B:D 合并不会自动重建。

4. **max_row 不缩减**：openpyxl 的 `max_row` 是已写入单元格的最高行号，`delete_rows` 删除后不会减小此值——即使行内容已清空。

5. **`_find_header_structure()` 仅扫描前50行**：该函数限制 `range(1, min(ws.max_row + 1, 50))`，对于数据行超过44行的明细表（如 4-8-4机器设备 151行原数据），合计行位置（R158）超出扫描范围，导致 `total_row=None`，`fill_sheet()` 无法识别结构。

**预防**：

1. **禁止裸 insert_rows/delete_rows 操作数据区域**：MUST 使用 `fill_sheet()` 接口。如需清空数据后重写，应通过 `_assert_idempotent_clear()` + `fill_sheet()` 的流程。

2. **对大数据量明细表的处理**：如果 `_find_header_structure()` 找不到合计行（total_row=None），原因可能是数据行数超过44行导致合计行超出前50行扫描范围。此时 MUST：
   - 手动定位合计行：直接扫描全表（`range(1, ws.max_row+1)`）找 A列="合计1"
   - 手动计算 data_start_row（通常=7）和合计行位置
   - 调用 `fill_sheet(ws=ws, sheet_name=sn, data_rows=..., wb=wb)` 时会在内部自动插行和格式修复
   - 或调用 `_apply_direct_format(ws, data_start, last_data)` 手动修复格式

3. **尾部空行处理**：`max_row` 尾部的空行不影响 Excel 打开，无需特殊处理。如需清理可调用 `smart_delete_rows()` 修复合并单元格和公式。

4. **序号修复**：写入数据后遍历数据行重新编号：`for i, r in enumerate(range(data_start, total1)): ws.cell(r, 2).value = i + 1`

**标准修复流程**（不依赖 fill_sheet 时）：
```python
# 1. 备份
# 2. 删除旧数据行（从 data_start 到 total1-1）
for _ in range(t1 - data_start):
    ws.delete_rows(data_start)
# 3. 插入新行
for _ in range(n):
    ws.insert_rows(t1)
# 4. 写入数据
for i, item in enumerate(items):
    r = data_start + i
    ws.cell(r, 2).value = i + 1  # 序号
    # ... 写入各列数据
    ws.cell(r, 11).value = item['cost']
    ws.cell(r, 12).value = item['net_value']
# 5. 修复合计行公式
ws.cell(t1, 12).value = f'=SUM(L{data_start}:L{last_data})'
# 6. 应用标准格式
from excel_row_ops import _apply_direct_format
_apply_direct_format(ws, data_start, last_data)
# 7. 设置行高
for r in range(data_start, last_data + 1):
    ws.row_dimensions[r].height = 15
# 8. 修复合计行 B:D 合并
for tr in [t1, bd, t2]:
    ws.merge_cells(start_row=tr, start_column=2, end_row=tr, end_column=4)
```

## 教训58：_find_header_structure()的50行扫描限制导致大数据量明细表合计行识别失败（邦能达项目2026-05复盘）

**问题**：`_find_header_structure(ws)` 函数仅扫描 `range(1, min(ws.max_row + 1, 50))` 即前50行。对于 `4-8-4机器设备` 等数据行数超过44行的明细表，合计行（R158）超出扫描范围，导致 `total_row=None`，后续 `fill_sheet()` 因无法找到合计行而写入失败。

**影响**：
- `fill_sheet()` 无法用于大数据量明细表的增补写入
- Agent 被迫使用裸 ws 操作 → 格式丢失

**根因**：该函数的50行上限是假设所有模板的数据行 ≤44行（A~F列共6个组）。但4-8-x固定资产子表可容纳数百项固定资产，合计行远超出50行。

**预防**：
1. 数据量超过44行时，MUST 手动扫描全表定位合计行
2. `_find_header_structure()` 应考虑提高扫描上限或动态计算
3. 在 `_find_header_structure()` 扫描不到合计行时，fallback 应全表扫描而非返回 None

**受影响项目**：邦能达2023评估明细表
