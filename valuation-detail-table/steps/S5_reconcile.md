## 操作

### Step 5.1 三级勾稽核对

```
明细表合计 → 汇总表 → 分类汇总表 → 资产负债表
   ↑              ↑          ↑              ↑

```

### Step 5.2 差异处理规则

| 差异类型 | 处理方式 |
|---------|---------|
| 科目余额表与BS一致，明细表与BS一致 | ✓ 无差异 |
| 科目余额表≠BS | 在明细表备注栏标注差异金额，写"科目余额表XX vs BS YY，差异ZZ待核实" |
| 明细表≠BS | 检查：是否有遗漏科目/行、SUM范围是否正确、减值准备是否填写 |
| 差异无法解释 | 在备注栏标注"待核实"，向用户报告差异情况 |

### Step 5.2a 科目归属交叉校验（DT-69 强制）

> **⚠️ 勾稽差异必须执行交叉校验，禁止仅做单科目检查。**

当某科目明细表合计≠BS时，MUST检查差异金额是否出现在其他Sheet中：

**典型交叉校验场景**：
- 长期借款差异55M → 检查其他应付款中是否有"结算中心借款"
- 预付款项差异150M → 检查是否含国土局/财政局大额（应重分类至其他非流动资产）
- 合同负债差异66M → 检查预收款项是否重分类
- 应付职工薪酬差异1.3M → 检查工会经费/职教费是否BS重分类至其他应付款

**执行方法**：收集所有Sheet合计值和各子项值，对每项差异在其他Sheet的子项中搜索金额匹配。

### Step 5.2b BS重分类处理决策树（DT-70 强制）

> **⚠️ 禁止将所有差异统一标注"BS重分类"而不具体说明。**

```
差异发现
  ├─ ① 差异金额=其他Sheet合计中的子项？
  │    └─ YES → 确认归属，标注"该金额归属XX科目，BS已重分类"
  │              如确认需调整→调整后重新勾稽
  │
  ├─ ② 差异金额=科目余额表与BS口径差异？
  │    └─ YES → 标注"科目余额表口径，BS已重分类至XX"
  │              保留明细表按科目余额表填写
  │
  └─ ③ 差异无法定位？
       └─ YES → 标注"待核实"+记录差异+继续执行（DT-151，全部完成后统一汇报）
```

### Step 5.2c 🚨 重分类映射表 [DT-118]

> **⚠️ 本步骤为v3.29新增硬约束，补充评估明细表填写的大逻辑框架**
> **核心概念**：重分类是指科目余额表中存在负数项目，编制资产负债表时将负数余额按性质重分类至对方科目。**重分类不是错误，是会计准则的编制要求。** 明细表合计MUST与BS一致（BS已包含重分类调整）。

**7种常见重分类关系（MUST逐一检查）**：

| # | 原科目(D2) | 负数方向 | 重分类去向(D1/明细表Sheet) | 典型场景 | 备注 |
|---|-----------|---------|--------------------------|---------|------|
| ① | 应交税费(2221) | 贷方负数（即借方余额） | 其他流动资产(3-4其他流动资产) | 待认证进项税额、预缴税费 | **已由DT-87覆盖**，MUST逐项填入其他流动资产 |
| ② | 应收账款(1122) | 贷方余额 | 预收款项(5-6预收款项) | 客户多付款、预收货款 | 1122贷方子项→5-6 |
| ③ | 预收款项(2203) | 借方余额 | 应收账款(3-5应收账款) | 客户退货、冲销预收 | 2203借方子项→3-5 |
| ④ | 应付账款(2202) | 借方余额 | 预付款项(3-7预付款项) | 供应商多付款、预付退回 | 2202借方子项→3-7 |
| ⑤ | 预付款项(1123) | 贷方余额 | 应付账款(5-5应付账款) | 收货未付款、供应商发票金额>预付 | 1123贷方子项→5-5 |
| ⑥ | 其他应收款(1221) | 贷方余额 | 其他应付款(5-10-3其他应付款) | 暂收款转为应付款 | 1221贷方子项→5-10-3 |
| ⑦ | 其他应付款(2241) | 借方余额 | 其他应收款(3-8-3其他应收款) | 垫付款转为应收款 | 2241借方子项→3-8-3 |

**执行规则**：

1. **Phase 0检测负数**：解析科目余额表时，MUST检测以下科目编码的所有子项的余额方向——2221/1122/2203/2202/1123/1221/2241。发现贷方负数或借方余额的子项→标记为重分类候选项
2. **Phase 1确认映射**：重分类候选项按上表映射到明细表对应Sheet，生成重分类映射表
3. **Phase 2按重分类填写**：重分类项填入目标Sheet（非原科目Sheet），在备注栏标注`[重分类]原科目XX贷方/借方负数YY，BS重分类至ZZ`
4. **Phase 4勾稽验证**：重分类后的明细表合计MUST与BS一致。不一致→检查是否有遗漏的重分类项

**重分类映射表格式**：

```markdown
## 重分类映射表

| # | 原科目编码 | 原科目名称 | 负数方向 | 金额 | 重分类至Sheet | 重分类至科目 | 备注 |
|---|-----------|-----------|---------|------|-------------|------------|------|
| 1 | 2221-1 | 待认证进项税额 | 贷方负数 | 3,456.78 | 3-4其他流动资产 | 其他流动资产 | [重分类]原科目2221-1贷方负数3,456.78，BS重分类至其他流动资产 |
| 2 | 1122-3 | XX公司 | 贷方余额 | 50,000.00 | 5-6预收款项 | 预收款项 | [重分类]原科目1122-3贷方余额50,000.00，BS重分类至预收款项 |
| ... | ... | ... | ... | ... | ... | ... | ... |
```

**与DT-70的关系**：DT-118是DT-70的细化——DT-70给出3级决策树（差异=其他Sheet子项/科目余额表口径差异/无法定位），DT-118具体列出7种重分类关系和执行规则。DT-70的①号判断路径中"差异金额=其他Sheet合计中的子项"，很多情况下就是重分类导致的。

### Step 5.3a 🚨 字段完整性校验（DT-161 L3兜底，上海图灵项目复盘新增）

> **⚠️ 本步骤为v3.49新增硬约束，上海图灵项目Phase 3被跳过后勾稽核对仍显示"通过"的复盘修复**
> **核心原则**：勾稽核对不仅验金额，还必须验字段完整性。往来科目的发生日期和业务内容为空时，勾稽结果应为WARN而非PASS。

**前置条件**：`file_manifest.json`中`序时账=True`。若`序时账=False`，跳过本步骤（DT-143）。

**执行步骤**：

```python
import openpyxl, json, os

# 1. 判断是否有序时账
cache_dir = os.path.join(PROJECT_DIR, '_dt_cache')
manifest_path = os.path.join(cache_dir, 'file_manifest.json')
has_journal = False
if os.path.exists(manifest_path):
    with open(manifest_path, 'r', encoding='utf-8') as f:
        manifest = json.load(f)
    for item in manifest.get('data', {}).get('files', []):
        fname = item.get('filename', '').lower()
        if '序时账' in fname or '明细账' in fname or 'journal' in fname:
            has_journal = True
            break

if not has_journal:
    print("Step 5.3a跳过: 无序时账(DT-143)，字段完整性校验不适用")
else:
    # 2. 扫描往来科目Sheet的发生日期和业务内容列
    wb_check = openpyxl.load_workbook(DETAIL_FILE, data_only=True)

    # 往来科目Sheet定义（同journal_extractor.py RECEIVABLE_SHEETS）
    receivable_sheets = {
        '3-5':  {'type': 'asset',     'date_col': 5, 'biz_col': 4, 'name': '应收账款'},
        '3-7':  {'type': 'asset',     'date_col': 5, 'biz_col': 4, 'name': '预付款项'},
        '3-8-3':{'type': 'asset',     'date_col': 5, 'biz_col': 4, 'name': '其他应收款'},
        '5-5':  {'type': 'liability', 'date_col': 4, 'biz_col': 5, 'name': '应付账款'},
        '5-6':  {'type': 'liability', 'date_col': 4, 'biz_col': 5, 'name': '预收款项'},
        '5-10-3':{'type':'liability', 'date_col': 4, 'biz_col': 5, 'name': '其他应付款'},
    }

    empty_date_count = 0
    empty_biz_count = 0
    generic_biz_count = 0
    total_rows = 0

    for sheet_key, cfg in receivable_sheets.items():
        # 查找实际Sheet名
        actual_name = None
        for sname in wb_check.sheetnames:
            if sheet_key.replace('-', '') in sname.replace('-', ''):
                actual_name = sname
                break
        if not actual_name:
            continue

        ws = wb_check[actual_name]
        if ws.sheet_state == 'hidden':
            continue

        # 扫描数据行
        for r in range(7, ws.max_row + 1):
            a_val = ws.cell(row=r, column=1).value
            if a_val and ('合' in str(a_val) or '减' in str(a_val) or '预' in str(a_val)):
                break
            name_val = ws.cell(row=r, column=3).value
            if not name_val:
                continue

            total_rows += 1
            date_val = ws.cell(row=r, column=cfg['date_col']).value
            biz_val = ws.cell(row=r, column=cfg['biz_col']).value

            if date_val is None or date_val == '':
                empty_date_count += 1
            biz_str = str(biz_val).strip() if biz_val else ''
            if not biz_str:
                empty_biz_count += 1
            elif biz_str in GENERIC_BIZ_CONTENTS:
                generic_biz_count += 1

    wb_check.close()

    # 3. 输出校验结果
    date_complete_rate = (total_rows - empty_date_count) / total_rows * 100 if total_rows > 0 else 100
    biz_complete_rate = (total_rows - empty_biz_count - generic_biz_count) / total_rows * 100 if total_rows > 0 else 100

    print(f"\n=== 字段完整性校验 ===")
    print(f"往来科目总数据行: {total_rows}")
    print(f"发生日期空: {empty_date_count}行 (完整率{date_complete_rate:.1f}%)")
    print(f"业务内容空/通用: {empty_biz_count + generic_biz_count}行 (完整率{biz_complete_rate:.1f}%)")

    if empty_date_count > 0 or empty_biz_count > 0:
        print(f"\n⚠️ WARNING: 字段完整性不足！")
        if empty_date_count > 0:
            print(f"  发生日期缺失{empty_date_count}行 → Phase 3可能未执行")
        if empty_biz_count > 0:
            print(f"  业务内容缺失{empty_biz_count}行 → Phase 3 Step 3.4可能未执行")
        print(f"  → 建议回退执行Phase 3（S3_journal_extract.md）")
    else:
        print(f"\n✅ 字段完整性校验通过: 发生日期和业务内容完整率100%")
```

**GENERIC_BIZ_CONTENTS定义**（与journal_extractor.py一致）：

```python
GENERIC_BIZ_CONTENTS = {
    '其他应收款', '其他应付款', '其他往来', '往来款',
    '销售商品/提供服务', '采购商品/接受服务', '预付货款/服务费', '预收货款/服务费',
    '货款', '预收账款', '应付货款',
}
```

**校验结果处理**：
- 完整率100% → PASS，进入Step 5.4
- 发生日期或业务内容有缺失 → WARN，但**不阻断**（Phase 6 Step 6.6a兜底拦截会最终阻断交付）
- 建议回退执行Phase 3，但不强制

### Step 5.4 🚨 差额推论映射 [DT-117]

> **⚠️ 本步骤为v3.27新增硬约束，河南平绿项目"其他货币资金"映射问题复盘修复**
> **根因**：科目余额表/资产负债表无对应科目编码的数据，在Phase 2被直接填入明细表，违反DT-0零幻觉原则。差额推论映射提供第三条路径：先完成直接映射填写→勾稽发现差额→差额推论确认映射→回填+标注。

**前置条件**：Step 5.1~4.3勾稽核对已完成，数据分类清单（Phase 0 Step 0.6）中有待确认映射数据。

**执行步骤**：

#### 4.4.1 收集差额

对每个含待确认映射数据的科目，计算直接映射部分的合计与BS的差异：

```
差额 = BS某科目期末余额 - 直接映射科目明细表合计

例：BS货币资金45,402,179.93 - (3-1-1现金0 + 3-1-2银行存款45,402,179.93 + 3-1-3其他货币资金0)
   = 45,402,179.93 - 45,402,179.93
   = 0
→ 勾稽通过，但BS子项"财务公司存款5,329,404.56"未在明细表反映
```

#### 4.4.2 差额匹配

将差额与待确认映射数据逐一比对：

```
推论映射决策树：
  │
  ├── 差额 = 某个待确认映射数据？
  │     └─ YES → 确认映射关系（见4.4.3）
  │
  ├── 差额 = 其他Sheet合计中的某个子项？
  │     └─ YES → 执行DT-69科目归属交叉校验
  │
  ├── BS有子项但明细表无对应分类？
  │     └─ 分析子项与待确认映射数据的关系
  │         └─ 子项金额≈待确认映射数据→推论映射
  │         └─ 子项金额≠待确认映射数据→标注"待核实"
  │
  └── 差额无法解释？
        └─ YES → 标注"待核实"+记录差异+继续执行（DT-151）
```

#### 4.4.3 确认映射并回填

差额匹配成功时，确认映射关系并回填明细表：

1. 将待确认映射数据填入对应Sheet
2. 在备注栏标注：`[推论映射] 金额XX来源于YY，因BS/科目余额表无对应编码，通过差额推论确认归属至ZZ科目`
3. 更新数据分类清单中该项的状态为"已确认映射"

**示例**：
```
3-1-3其他货币资金，保证金5,870,945.58：
- 推论依据：BS货币资金中"财务公司存款5,329,404.56"与保证金中的建行保证金部分对应，
  推论保证金属于"其他货币资金"分类，BS将其归入"财务公司存款"子项
- 备注：[推论映射] 金额5,870,945.58来源于保证金对账单PDF(6份)，
  因科目余额表/BS无1012其他货币资金编码，通过BS子项分析推论确认归属
```

#### 4.4.4 未匹配项处理

差额无法匹配任何待确认映射数据时：
- 在明细表备注栏标注"待核实"
- 在数据分类清单中该项状态保持"待确认映射"
- 输出WARNING+标注[待核实]+继续执行（DT-151）
- **gate_validator.py G2-10**：Phase 4完成后仍存在"待确认映射"状态项=CRITICAL=禁止交付
